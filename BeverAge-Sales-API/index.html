    Documentação da Beverage Sales API¶

    Visão Geral¶

    A Beverage Sales API é uma aplicação baseada em FastAPI para expor um serviço de inferência de machine learning com foco em previsão de vendas e sinais de anomalia.

    Com base nos arquivos enviados, a API utiliza:

    • um modelo principal de predição carregado com joblib
    • um modelo de anomalia baseado em Isolation Forest
    • metadados do modelo carregados de um arquivo JSON
    • dados históricos em formato parquet
    • uma etapa de engenharia de atributos para preparar a entrada antes da predição

    Objetivo da API¶

    O objetivo é receber uma nova observação de venda, localizar um histórico compatível por Product e Region, gerar features com base nesse histórico, calcular sinais de anomalia e então produzir a predição final.

    Em termos práticos, a API segue uma arquitetura de inferência híbrida:

    1. valida o payload recebido
    2. carrega o histórico de referência
    3. gera features temporais e agregadas
    4. calcula variáveis auxiliares para anomalia
    5. executa o modelo final de predição
    6. devolve o resultado em JSON

    Data¶

    Parquet file¶

    In the Beverage-Sales project, the notebook 04_anomaly_detection.ipynb generates the compressed Parquet file: 📦 anomaly_predictions_api.parquet

    Inside the API repository, this file must be placed in the following directory: app/data

    To ensure the API can access this data, references need to be updated in two configuration files (which has already been set up in this version): 🛠️ app/src/config/src_config.py 🛠️ app/core/config.py
    --- --- ---
    No projeto Beverage-Sales, o notebook 04_anomaly_detection.ipynb gera o arquivo Parquet compactado: 📦 anomaly_predictions_api.parquet

    No projeto da API, este arquivo deve ser armazenado no diretório: app/data

    Para que a API consuma esses dados, é necessário atualizar o caminho em dois arquivos de configuração (o que já foi implementado nesta versão): 🛠️ app/src/config/src_config.py 🛠️ app/core/config.py

    Endpoints Identificados¶

    GET /health¶

    Endpoint simples para verificação de disponibilidade da API.

    Resposta esperada:

    {
  "status": "ok"
}

    POST /login¶

    Endpoint de autenticação.

    Recebe:

    • username
    • password

    Se as credenciais forem válidas, retorna um token de acesso do tipo bearer.

    Resposta esperada:

    {
  "access_token": "<token>",
  "token_type": "bearer"
}

    POST /predict¶

    Endpoint principal de predição.

    Este endpoint exige autenticação e recebe um payload com dados de pedido, cliente, produto, preço, quantidade, desconto, região e data do pedido.

    Campos esperados:

    • Order_ID
    • Customer_ID
    • Customer_Type
    • Product
    • Category
    • Unit_Price
    • Quantity
    • Discount
    • Total_Price
    • Region
    • Order_Date

    Fluxo interno do endpoint /predict¶

    1. O payload é convertido para DataFrame.
    2. Os campos numéricos e de data são normalizados.
    3. O histórico parquet é carregado.
    4. O histórico é filtrado por Product e Region.
    5. Os dados históricos e o novo registro são combinados.
    6. A classe DataQualityFeatures gera as features.
    7. A linha correspondente à data de predição é selecionada.
    8. São criadas features auxiliares para o modelo de anomalia.
    9. O Isolation Forest gera:
    10. anomaly_flag
    11. anomaly_score
    12. anomaly_label
    13. O modelo LightGBM executa a predição final.
    14. A API retorna o valor previsto e informações adicionais.

    Exemplo de resposta¶

    {
  "status": "success",
  "prediction": 134.27,
  "target": "quantity_sum",
  "model_version": "v1",
  "model_name": "Beverage Sales Hybrid Forecasting Model",
  "anomaly": {
    "anomaly_flag": 0,
    "anomaly_score": 0.1842,
    "anomaly_label": "Normal"
  },
  "input_reference": {
    "Order_ID": "ORD-1001",
    "Product": "Cola 350ml",
    "Region": "Baden-Württemberg",
    "Order_Date": "2023-11-10"
  }
}

    Endpoints de Debug¶

    Os arquivos enviados também mostram uma área de debug da API, útil para validação de ambiente e checagem dos recursos carregados.

    GET /debug/paths¶

    Verifica se os principais arquivos existem.

    Inclui checagem para:

    • metadata
    • parquet
    • modelo LightGBM
    • modelo de anomalia

    GET /debug/metadata¶

    Carrega e resume os metadados do modelo.

    Retorna, entre outros:

    • status
    • chaves do metadata
    • target
    • versão do modelo
    • quantidade de raw features
    • quantidade de model features

    GET /debug/parquet-sample¶

    Carrega uma pequena amostra do parquet histórico.

    Retorna:

    • status
    • número de linhas
    • colunas
    • registros de exemplo

    GET /debug/models¶

    Verifica se os modelos conseguem ser carregados.

    Retorna os tipos dos objetos carregados.

    GET /debug/validate-schema¶

    Compara as colunas do parquet com as features esperadas nos metadados.

    Retorna:

    • quantidade de colunas no parquet
    • quantidade de raw features
    • quantidade de model features
    • raw features ausentes
    • model features ausentes

    GET /debug/full-check¶

    Executa uma validação completa dos recursos principais da API.

    Verifica ao mesmo tempo:

    • metadata
    • parquet
    • modelo LightGBM
    • modelo de anomalia

    Dependências Funcionais Observadas¶

    Com base nos arquivos enviados, a API depende de alguns componentes internos que não foram incluídos no material:

    • app.core.security
    • app.src.config.src_config
    • app.src.features.data_quality_features

    Isso significa que parte do comportamento foi documentada com base no uso observado no código.

    Recursos de Arquivo Esperados¶

    A API espera a existência de arquivos locais, com paths vindos de src_config, como por exemplo:

    • arquivo do modelo LightGBM
    • arquivo do modelo de anomalia
    • arquivo de metadados JSON
    • parquet histórico

    Sem esses arquivos, a API pode responder com erro interno.

    Tratamento de Erros¶

    O endpoint /predict trata os seguintes cenários:

    • erro de validação de dados → HTTP 400
    • arquivo ausente → HTTP 500
    • erro inesperado → HTTP 500

    Além disso, se não houver histórico para o par Product + Region, a API retorna erro de validação.
