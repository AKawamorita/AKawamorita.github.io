    Endpoint Details¶

    This document describes the endpoints identified in the uploaded FastAPI route files for the Beverage Sales API.

    1. Health Endpoint¶

    GET /health¶

    Checks whether the API is running.

    Purpose - quick operational validation - useful for containers, probes, and monitoring

    Success response

    {
  "status": "ok"
}

    Authentication required - No

    2. Authentication Endpoint¶

    POST /login¶

    Authenticates a user and returns an access token.

    Request body

    {
  "username": "your_user",
  "password": "your_password"
}

    Success response

    {
  "access_token": "<token>",
  "token_type": "bearer"
}

    Error response - 401 Unauthorized when credentials are invalid

    Authentication required - No

    Notes - The token is created with an expiration window of 60 minutes according to the uploaded code. - The implementation depends on internal security functions: - authenticate_user - create_access_token

    3. Prediction Endpoint¶

    POST /predict¶

    Runs the main prediction flow using historical data, derived features, anomaly signals, and the final LightGBM model.

    Authentication required - Yes

    The route uses a dependency called get_current_user, so the request must include a valid bearer token.

    Request body¶

    {
  "Order_ID": "ORD-1001",
  "Customer_ID": "CUST-10",
  "Customer_Type": "Retail",
  "Product": "Cola 350ml",
  "Category": "Soft Drinks",
  "Unit_Price": 4.50,
  "Quantity": 20,
  "Discount": 0.10,
  "Total_Price": 81.00,
  "Region": "Baden-Württemberg",
  "Order_Date": "2023-11-10"
}

    Field description¶

    Field Type Description
    Order_ID string Order identifier
    Customer_ID string Customer identifier
    Customer_Type string Customer segment or type
    Product string Product name
    Category string Product category
    Unit_Price float Unit price value
    Quantity float Quantity value
    Discount float Discount value
    Total_Price float Total order price
    Region string Sales region
    Order_Date string Order date, converted to datetime

    Internal processing steps¶

    The uploaded code shows the following execution path:

    1. request payload is validated by PredictRequest
    2. payload is converted to a DataFrame
    3. numeric fields are coerced to numeric values
    4. Order_Date is parsed as datetime
    5. parquet historical data is loaded
    6. history is filtered by Product and Region
    7. historical data is combined with the new row
    8. DataQualityFeatures builds the feature set
    9. the row for the requested prediction date is selected
    10. anomaly helper features are created:
    11. if_qty_signal
    12. if_sales_signal
    13. if_discount_signal
    14. if_ticket_signal
    15. Isolation Forest generates:
    16. anomaly_flag
    17. anomaly_label
    18. anomaly_score
    19. LightGBM predicts the final output

    Success response example¶

    {
  "status": "success",
  "prediction": 134.27,
  "target": "quantity_sum",
  "model_version": "v1",
  "model_name": "Beverage Sales Hybrid Forecasting Model",
  "anomaly": {
    "anomaly_flag": 0,
    "anomaly_score": 0.1842,
    "anomaly_label": "Normal"
  },
  "input_reference": {
    "Order_ID": "ORD-1001",
    "Product": "Cola 350ml",
    "Region": "Baden-Württemberg",
    "Order_Date": "2023-11-10"
  }
}

    Possible error cases¶

    400 Bad Request¶

    Returned for validation or business-rule problems, such as:

    • invalid Order_Date
    • missing compatible history for the same Product and Region
    • failure to build the prediction feature row

    500 Internal Server Error¶

    Returned for problems such as:

    • missing model file
    • missing parquet file
    • missing metadata file
    • unexpected internal exception

    4. Debug Endpoints¶

    These endpoints help validate local resources and inspect the project state.

    GET /debug/paths¶

    Checks file existence for key resources.

    Returns - metadata path and existence - parquet path and existence - LightGBM model path and existence - anomaly model path and existence

    Authentication required - Not shown in the uploaded code

    GET /debug/metadata¶

    Loads metadata JSON and returns a summary.

    Returns - status - metadata keys - target - model version - raw feature count - model feature count

    Authentication required - Not shown in the uploaded code

    GET /debug/parquet-sample¶

    Loads a small parquet sample.

    Returns - status - row count - column names - sample records

    Authentication required - Not shown in the uploaded code

    GET /debug/models¶

    Attempts to load both model files.

    Returns - status - LightGBM model Python type - anomaly model Python type

    Authentication required - Not shown in the uploaded code

    GET /debug/validate-schema¶

    Compares metadata feature definitions against parquet columns.

    Returns - parquet column count - raw feature count - model feature count - missing raw features - missing model features

    Authentication required - Not shown in the uploaded code

    GET /debug/full-check¶

    Runs a combined validation of metadata, parquet, and models.

    Returns - status - resource load flags - metadata keys - sample columns - sample record - model object types

    Authentication required - Not shown in the uploaded code
