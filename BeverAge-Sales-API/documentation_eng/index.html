    Beverage Sales API Documentation¶

    Overview¶

    The Beverage Sales API is a FastAPI application designed to expose a machine learning inference service for sales prediction with anomaly-aware logic.

    Based on the provided files, the API uses:

    • a prediction model loaded with joblib
    • an anomaly detection model based on Isolation Forest
    • model metadata loaded from JSON
    • historical reference data stored in parquet format
    • a feature engineering step before the final inference

    Main Objective¶

    The API receives a new sales-related record, finds compatible historical data by Product and Region, creates derived features, adds anomaly signals, and returns the final prediction.

    This creates a hybrid inference flow with:

    1. payload validation
    2. historical reference loading
    3. feature generation
    4. anomaly signal generation
    5. final prediction
    6. structured JSON response

    Identified Endpoints¶

    GET /health¶

    Simple health check endpoint.

    Response

    {
  "status": "ok"
}

    POST /login¶

    Authentication endpoint.

    It receives:

    • username
    • password

    If authentication is successful, it returns a bearer token.

    Response

    {
  "access_token": "<token>",
  "token_type": "bearer"
}

    POST /predict¶

    Main secured prediction endpoint.

    This route requires authentication and receives a structured sales request with order, customer, product, pricing, quantity, region, and date data.

    Expected fields:

    • Order_ID
    • Customer_ID
    • Customer_Type
    • Product
    • Category
    • Unit_Price
    • Quantity
    • Discount
    • Total_Price
    • Region
    • Order_Date

    Internal Prediction Flow¶

    1. The request payload is converted into a pandas DataFrame.
    2. Numeric and date fields are normalized.
    3. Historical parquet data is loaded.
    4. The reference history is filtered by Product and Region.
    5. Historical data and the new record are concatenated.
    6. DataQualityFeatures is used to build the derived features.
    7. The feature row for the prediction date is selected.
    8. Auxiliary anomaly features are created.
    9. The Isolation Forest model generates:
    10. anomaly_flag
    11. anomaly_score
    12. anomaly_label
    13. The LightGBM model performs the final prediction.
    14. The API returns the response in JSON format.

    Example Prediction Response¶

    {
  "status": "success",
  "prediction": 134.27,
  "target": "quantity_sum",
  "model_version": "v1",
  "model_name": "Beverage Sales Hybrid Forecasting Model",
  "anomaly": {
    "anomaly_flag": 0,
    "anomaly_score": 0.1842,
    "anomaly_label": "Normal"
  },
  "input_reference": {
    "Order_ID": "ORD-1001",
    "Product": "Cola 350ml",
    "Region": "Baden-Württemberg",
    "Order_Date": "2023-11-10"
  }
}

    Debug Endpoints¶

    The uploaded files also expose a debug router for environment validation and resource inspection.

    GET /debug/paths¶

    Checks whether the main project files exist.

    It validates:

    • metadata file
    • parquet file
    • LightGBM model file
    • anomaly model file

    GET /debug/metadata¶

    Loads and summarizes the metadata file.

    Returns values such as:

    • status
    • metadata keys
    • target
    • model version
    • number of raw features
    • number of model features

    GET /debug/parquet-sample¶

    Loads a small sample from the parquet dataset.

    Returns:

    • status
    • row count
    • column list
    • sample records

    GET /debug/models¶

    Checks whether the models can be loaded correctly.

    Returns the Python object types for:

    • LightGBM model
    • anomaly model

    GET /debug/validate-schema¶

    Compares parquet columns with the expected raw and model features from metadata.

    Returns:

    • parquet column count
    • raw feature count
    • model feature count
    • missing raw features
    • missing model features

    GET /debug/full-check¶

    Runs a combined validation for all major resources.

    It attempts to load:

    • metadata
    • parquet data
    • LightGBM model
    • anomaly model

    Referenced Internal Modules¶

    Some modules are referenced in the uploaded files but were not included in the shared material:

    • app.core.security
    • app.src.config.src_config
    • app.src.features.data_quality_features

    Because of that, part of this documentation was inferred from the way these modules are used in the routes.

    Expected Local Resources¶

    The API depends on local files configured through src_config, including:

    • LightGBM model file
    • Isolation Forest model file
    • metadata JSON file
    • historical parquet dataset

    If any of these resources is missing, the API may return internal errors.

    Error Handling¶

    The /predict endpoint explicitly handles:

    • invalid input or business validation problems → HTTP 400
    • missing required files → HTTP 500
    • unexpected internal failures → HTTP 500

    The route also returns a validation error when no historical data is found for the same Product and Region.
